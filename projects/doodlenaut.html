Doodlenaut

A character customization flow where players shape a character’s personality. We compared two interaction styles—trait selection vs sliders—and measured usability, satisfaction, and time-on-task.

Overview

What it is, who it’s for, and what we tested.

Product

  • Web-based customization flow (landing → optional quiz → customization → completion).
  • Designed to be fun + simple for kids ages 6–12.
  • Built in React with UI styling in CSS.

Research question

  • Does a simpler “pick traits” UI feel more usable than a more precise “slider” UI?
  • We measured SUS, ease, satisfaction, and completion time.
  • Mixed-methods: quant metrics + qualitative feedback/themes.

TL;DR

Users preferred the slider UI for clarity + control (higher SUS/ease/satisfaction), even though the trait grid was faster.

Problem

Why this matters from an HCI perspective.

Design tension

Speed vs control

In customization, “fast choices” can reduce friction, but “fine control” can increase confidence and satisfaction. We wanted to learn which interaction style better supports the user’s mental model.

Choice architecture • Feedback • Learnability

Success criteria

Measurable

A successful design should feel easy to use, help users understand what their choices mean, and reduce uncertainty (especially for younger users).

SUS • Ease • Satisfaction • Time-on-task

Design variations

Two ways to customize personality traits.

Design A — Trait grid

Lower complexity

Users select any 5 traits out of 10 from a grid. Selected traits appear in-order and can be removed.

Selection limit • Add/remove traits • “Done” to finish

Design B — Sliders

Higher precision

Users adjust sliders between opposing traits (e.g., Creative ↔ Analytical) to fine-tune personality.

Granular control • Adjust anytime • “Done” to finish

Method

How we ran the A/B comparison and what we captured.

1) Task
Customize a character’s personality, then finish the flow.
2) A/B exposure
Participants used either the trait grid (A) or slider UI (B).
3) Metrics
SUS, ease (1–7), satisfaction (1–7), completion time, plus interview notes.
4) Analysis
Quantitative analysis in R + qualitative coding/themes in MAXQDA.

What I’d highlight in a grad review

This wasn’t just “which UI is prettier?”—it tested how interaction style changes perceived usability and confidence, using mixed methods to connect numbers to user quotes/themes.

Results

Key metrics + the story they tell.

SUS (avg)
A: 75 → B: 82
B rated more usable overall
Ease (1–7)
A: 5.4 → B: 6.3
B felt easier to customize
Completion time
A: 4.2m → B: 5.1m
A was faster; B took longer
Satisfaction (1–7)
A: 5.6 → B: 6.1
B slightly more satisfying

Qualitative themes

  • Users liked the art style + “fun” feel, but wanted clearer instructions.
  • Some features were confusing without guidance (quiz, sliders for some users).
  • Takeaway: speed isn’t everything—users preferred control + clarity.

So what?

The sliders increased perceived usability even with longer completion time—suggesting that for customization, confidence and clarity can matter more than speed.

Design implications

Concrete changes I’d make based on what we learned.

Make meaning visible

Reduce uncertainty

Add microcopy and previews (“You’re leaning more creative than analytical”) leading to a stronger mental model.

Feedback • Interpretation • Learnability

Progress + guidance

Lower cognitive load

Add a simple progress indicator and “What do I do?” helper for first-time users, especially younger players.

Onboarding • Visibility of system status

Hybrid approach

Best of both

Start with trait chips (quick picks), then allow optional sliders for fine-tuning—supporting both speed and control.

Progressive disclosure

Accessibility considerations

Inclusive UX

Ensure keyboard + screen reader support for sliders and clearly labeled endpoints (not just color/position).

Focus states • Labels • Contrast

Reflection

What I’d do next if this were a real product.

Limitations

  • Would benefit from a larger sample and more representative age distribution.
  • Sliders may require extra onboarding for younger users.

Next steps

  • Test a hybrid UI (chips → optional sliders) with progressive disclosure.
  • Add clearer instructions + a “preview personality” summary card.
  • Evaluate accessibility with keyboard-only + screen reader checks.
